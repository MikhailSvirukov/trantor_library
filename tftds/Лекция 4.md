* Atomic Broadcast (если 2 узла делали операцию broadcast, то есть гарантии, что обработчики на узлах будут вызваны в одном и том же порядке + свойства: agreement (все несбойные узлы получат одни и те же сообщения), validity (все узлы получат все сообщения), total order (порядок вызова обработчиков))
* RSM(apply - применить операцию)
* Задача Консенсуса (У каждого узла есть начальное значение, и они должны реализовать propose (только из предложенных, лжно и то же значение, всегда завершается)) (validity, agreement, termination)
* Серия консенсусов для определения порядка: на каждом шаге - выбор сообщения. (все это после reliable broadcast)

*Насколько хорошо решаема задача консенсуса?*
* отказоустойчивость (ограничена половиной числа узлов для реплицируемого регистра) 
	* если алгоритм завершается для произвольного partition с двух сторон, то он нарушает свойство  agreement (одинаковый порядок сообщений), то есть он должен блокироваться для какой-то части partition (для любого partition должен переживать смерть только одной из половин, причем большой, потому что иначе переживает отказ меньше половины узлов); узел блокируется навсегда в меньшей partition
	* алгоритм не отличает отказы от partition
	* невозможно лучше половины
	* чтобы понять число, алгоритм собирает кворумы
	* при любом числе отказов невозможно одновременно добиться выполнения termination и agreement
* асинхронный мир; детерминированный алгоритм для консенсуса; отказов не больше 1 - задача оказывается неразрешима (теорема FLP - в асинхронной модели невозможно добиться termination )
*  что если перейти в частично-асинхронную модель (преодолеть отсутствие termination в FLP)? Все ок, это буквально таймауты
* построение детекторов сбоев (в каких предположениях о времени можно построить - должен выбрать лидера)
