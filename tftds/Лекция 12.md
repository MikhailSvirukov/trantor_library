### Распределенные транзакции

Google Spanner
* щардированное распределенное key-value storage
* shared хранилища - 3 реплики в разных Data Center, репликация через MultiPaxos (есть стабильный лидер), хранит данные tablet, под shared - распределенная файловая система
* транзакции задевают много shared
* протокол транзакций через 2PL c Wound-Wait
* но есть отказы - клиента и shared; 
* **фомально мы пользуемся допущением, что мы не берем сразу exclusive блокировки на чтение - мы копим их, и отправляем вместе с сообщением о коммите (при этом мы продолжаем работать по протоколу 2PL)**
* для начала рассмотрим ситуацию, когда транзакции задевают только ключи из одного shared (и соответственно, определенного набора реплик);
	* 2PL в такой картине:
		* все блокировки, ассоциированные с ключами хранятся в lock table, обслуживается эта таблица  через transaction manager (один на shared); блокировки и менеджер хранятся в лидере, при этом не persistent - при перезагрузке блокировки не сохраняются
			* если умирает клиент, который держит бдокировку - решение: для транзакции устанавливается сессия (таймер); клиент посылает heartbeat; если ответа нет - транзакция отзывается, блокировки снимаются, при этом промежуточные результаты операций которые были выполнены не сохраняются persistent
			* если умирает лидер (перевыюран) - сгорят все сессии, соответственно блокировки исчезнут, просто клиенты будут делать retry
* если транзакция пользуется данными нескольких shared:
	* проблема: при рассылке коммита сообщения идут к нескольким shared, каждый из которых самостоятельно принимает решение о коммите своей порции; как им договариваться? **Atomic Commit problem**
* **Atomic Commit**
	* resource manager, который локально принимает решение о коммите
	* нужно откатить транзакцию, если хотя бы один resource manager (shared) против, и принять коммит иначе
	* **2 Phase-Commit** (2PC)
		* узел координатор + resource managers (много)
		* координатор в первой фазе шлет сообщение prepare
		* resource manager шлет ответ (да, если готов и гарантирует, что закоммитит)
		* если все ок - коммитим (шлем сообщение), если хотя бы один нет - abort
	* в Spanner Commit Coordinator - клиент; при получение prepare блокировки персистентно сохраняются  shared; потом коммит (если выше - см выше)
		* проблем с падением shared нет
		* но если после prepare клиент умер - вечные блокировки
		* с эти можно боротся, если занести координатора внутрь системы
	* альтернативное решение:
		* клиент выбибиарет среди используемых  shared лидера (посылает вместе с prepare номер коррдинатора); все остальные отвечают на prepare именно этому shared; если координатор перезагрузится - не страшно, он отказоустойчив
		* проблема: если клиент не успел отправить всем prepare; координатор уже может использовать таймаут
		* то есть 3 надежные записи: блокировок на shared, решение на координаторе + транзакции на shared
	* 2PL - для Isolation (про много транзакций) и 2PC - Отказоустойчивость (про одну транзакцию)
* Оптимизации:
	* read-only transaction - Snapshot Isolation + read+write transaction - 2PL
		* нужно согласовывать порядок операций - т.к. в 2PL порядок от блокировок, а в SI - порядок временных меток
		* выбирать монотонные интервалы времени для блокировки: timestamp Oracle - просто RSM (плохая пропускная способность); можно собирать запросы в интервале времени и выдавать всем запросам метоку StartTime+AmountRequests
		* TrueTime - не хочется одну глобальную точку (даже отказоустойчивую и многообъемную) - долго ходить; ответ - интервал 
		* Время обновляется раз в 30 секунд у мастера в рамках дата центра и работает с учетом дрейфа 
		* тогда в качестве метки - правая граница интервала true time
		* вообще ответственность на выборе лежит на клиенте - он через API сообщает, что он будет/не будет писать;
		* чтобы тарнзакция на чтение с самой новой меткой относительно всем закоммиченным транзакциям - нужно просить у TrueTime на машине со spanner время; а если еще не все дошли предыдущие? shared ждет интервал safe time, на который откладывает пришедшие операции;
