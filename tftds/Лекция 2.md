### Replication
* Как отказоустойчиво хранить данные, которые помещаются в одну машину?
* *Задача*: репликация регистра (операции записать и прочитать)
* В каждый момент времени в репликах может быть разные значения этой ячейки (скорость света в проводах)
* Реализация генерирует истории операций в системе; ожидаем ли что-то касательно результатов операций?
* гарантии: записанное значение будет прочитано (запись не теряется); или показывать всем пользователям все записи (события); как показать системе причинность?
* цепочка операций - последовательная история (в противовес конкурентной); 
* реализация порождает конкурентные: модель согласованности - какие конкурентные истории может порождать реализация для заданной спецификации (последовательная история)
* Linearizability (external consistency): для любой конкурентной истории для регистра, существует конкурентная история, такая что: те де операции с теми же результатами + если операция завершилась раньше, чем началась вторая, то в последовательной истории операции идут в таком порядке (Real Time ordering); то есть если операции пересекаются во времени, то между ними нет отношения happens before - пользователь не ожидает между ними какой-то упорядоченности
* реализация реализуема - если любая история лианеризуема (тогда пользователь может не думать о том, что она распределенная) (атомарность для регистра)
* как понять порядок операций, если часы не синхронны? Иногда пользователи знают порядок, думают про happens before; система такое отношение не знает; как понять какая запись раньше? есть timestamps; два порядка - happens before/timestamps - надо согласовать; Real Time ordering - согласует 
#### Реализация Atomic Register
* write/read, async, crash/restart (взрыв)
* сколько машин для отказоустойчивости? 3
* статическая конфигурация машин; клиент знает номера машин и их число; только 1 клиент - допущения для упрощения
* запись машин: когда операция завершилась? - клиент хочет подтверждения? - сколько? не 1 - не отказоустойчиво, не 3 - много; значит 2 подтверждения; новая запись .... 
* как синхронизировать записи внутри системы? - клиент снабжает записи временными метками; то есть реплика не перезаписывает запись более новую на более старую (но подтверждение всегда есть)
* как делать чтение? у реплик разное значение; каждые 2 хранят правильное значение - запросы на 3 - ждем 2; берем по наибольшему timestamp
* для 3 машин - не более 1 отказа система переживает
* обобщение: quorum system (у каждого набора таких узлов, которые подтверждают две соседние операции есть непустое пересечение; корум на чтение пересекается с кворумом на запись)-> majorities (для n узлов - n/2 отказов)
* сколько реплик тогда выбирать? нечетное - растет кворум, но не увеличивается отказоустойчивость (вообще можно кворумы регулировать по численности на конкретную операцию)
* если отказов больше, чем можно? система перестанет отвечать
* вообще, алгоритм не был правилен, поскольку нет каких-то выборов для поддержки условий линеаризуемости (не различает пересечение с записью - еще происходит или уже не произошел)
* Алгоритм: если в кворуме 2 одинаковых значения - все ок, если разные - запишем причем собираем полноценный кворум! (то есть сразу после чтения - запись, причем клиент о записи-обновлении не знает); причем если чтение затянулось и следующее чтение может вернуть предыдущее значение - но про пересекающиеся операции мы не даем гарантии
