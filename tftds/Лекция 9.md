### DFS
* хранит сколь угодно болшие файлы + йерархия имен
* inode для каждого файла + блоки (по ним поиск на блочном устройство)
* два логических уровня: данные и метаданные - разделить в две подсистемы - MetaStore и ChunkStore 
* только append - добавление нового чанка (без перезавписи, только пишем в конец), а потом обновление метаданных
* Для хранения чанков - пулл машин; брали chunkmaster - выбирала для запроса несколько машин из пула и писала на них реплики
* менеджер записи не должен помнить, на каких машинах ледат какие чанки; машина сообщает о своих чанках всем чанкмастерам; явного консенсуса не нужно - потому что монотонно растущее множество; тоже шардируемся по shard store - горизонтальная масштабируемость без задачи консенсуса (во многом из-за иммутабельности данных)
* для метаданных проблемы - много изменений и важен порядок crud - поэтому RSM с paxos; ограничена масштабируемость тем, что RSM живет на одной машине; шардировать дерево - сложно; решение - перложить в хранилище ключ-значение (описана некоторая схема)
* такой подход исключает атомарность - разные данные нескольких директорий на разных машинах; но в рамках таблетов можно делать атомарность (над одним файлом, например)

### Big Table
* сделать LSM поверх распределенной файловой системы (до этого мы делали LSM чтобы избежать отказа одного диска, а DFS - чтобы отказа без рестарта)
* был слой репликации над хранилищем, но хранилище - муталельно; поэтому сложное изменение; если перевернуть - получается репликация иммутабельных данных; 
* логическое разделение хранения и обслуживание; для каждого таблета - дна машина - при смерти не нужно перемещать данные таблеты
* но проблема - при инверсии мы хотим чтобы big table использовало dfs, а dfs хранит метаданные в big table - цикл
* задача: промасштабировать файловую систему через KVS, которое требует в зависимости файловую систему - нужно сделать *взамную рекурсию*
* но это прямо жесть - долго спускаться по дереву рекурсии
* хочется уметь перезаписывать, чанки, не добавляя метаданные: у каждого чанка единственный писатель (если меняется - чанк закрывается)

Итого, теперь на единой машине находится и координатор, и сами LSM, которые представляют лог; почему безопасно, если машина умрет?
* спользуем распределенные блокировки
* для борьбы со случаями, когда 2 узла считают, что владеют блокировками (например, произошла блокировка на сборку мусора и машина на это время подвисла); по сути, это подъем задачи консенсуса на уровень выше (на уровне файловой системы) - IO fencing
* tablet server (обслуживает tablet) берет блокировку в большой таблице - а пишет данные в файловую систему (внешняя система); файловой системе нужно защита от второго лидера - механизм lease (если появляется второй, то первый инвалидируется или переименование директории)
