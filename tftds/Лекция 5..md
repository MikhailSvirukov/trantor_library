## Single- decrease Paxos
* По сути - решение задачи принятия одного указа
* Consensus problem: n узлов, каждый реализует propose (получает input и выбирает общее значение (из предложенных), выбор одинаковый, алгоритм завершается)
* в sds - на узлах появляются роли: proposer (непосредственно предлагает значение ) + acceptor (голосует за предложение, но может менять выбор); предложение - пара занчение + proposal number (глобально разные, обычно монотонны для каждого proposer)
* обычно proposal number - (k, id proposer)
* для данного алгоритма не важдо число proposer, но число acceptors фиксировано, при этом в алгоритме это могут быть разные узлы 
* коммункация: proposer общается только с acceptor, acceptor только отвечает на сообщения
* как сделать общий выбор?  по сути proposer делает lock на всех acceptor, и делает предложение как раз снимая эти locks; только один proposer может собрать кворуум
* но эти блокировки можно отбирать, скорее lease; то есть proposer может перехватывать блокировки
* алгоритм propose: 
	1. выбрал propose number
	2. шлет всем acceptor сообщение prepare (с аргументом pn); каждый acceptor хранит max propose number (самый большой полученный prepare); prepare отвергается если число в запросе меньше MPN; иначе - обновление MPN и отвечает proposer сообщением promise (аргументы - старое число + пара (текущий отданный acceptor голос, за что он пока (уже) проголосовал)); 
	3. proposer собирает кворуум
	4. после того, как собрал - смотрит на значения, которые собрал и выбирает значение для реального предложения; если proposer получил хотя бы один непустой голос, то в качестве предложения будет сообшение с максимальным числом (из той пары которую возвращает acceptor); если все пустые - то proposer выбирает свой собственное сообщение 
	5. отправление команды propose_n_v (accept) для acceptor (аргументы команды - собственное число + сообщение)
	6. acceptor получает команду accept и меняет значение сообщения, если число в запросе не меньше того, что во внутреннем состоянии (именно MNP); отвечает сообщением accepted (аргумент - число)
	7. при этом ждем кворуум и на acceptor
	8. в случае неудач - retry c большим числом (acceptor может отвечать со своим числом, чтобы proposer мог выбрать побольше число)
	9. если итерация прошла успешно (собраны оба кворуума), то proposer выбирает сообщение, которое предложил

	Замечание:
	* acceptor обновляет свой MPN только на обоих этапах (команда accept), плюс   на втором этапе обновляется пара текущего голоса (число + сообщение) (опять же, если сообщение принимается)
	* Данный алгоритм только для одного выбора, а не последовательности - proposer всегда в accept отправляет полученное сообщение из promise, если оно непустое
	* То есть promise нужен для определения сообщения accepted
