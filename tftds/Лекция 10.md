***TLA+***
* как форма документации + можно писать программу для механической системы (тупо перечислить все состояния системы)
* как устроен:
	* есть система и ее свойства
	* в FLP - граф конфигурации вместо конкретного алгоритма; покрывает все пути исполнения
	* *свойства*: safety (mutual exclusion -  про каждую вершину графа ) + liveness (deadlock freedom - про каждую траекторию в графе)
	* конфигурации - State с какими-то значениями переменных; нужно уметь делать утверждения о значениях утверждения; выбрать формальную логику - **Логика первого порядка** - но этого хватает только для свойств safety; для liveness - **темпоральная логик**а (траектория - цепочка состояний - Trace) 
		* темпоральные операторы (где после оператора -  обычная формула):
			* always (формула выполняется в каждом состоянии траектории - invariant)
			* eventually (в какой-то конфигурации траектории формула выпонится)
			* next (следующий выполнится)
		* траектория A выполняет формулу f, если первая конфигурация траетории выполняет формулу f (если формула без темпоральных операторов)
		* траектория A выполняет формулу always f - если любой суффикс траектории выполняет f
		* траектория A выполняет формулу eventually f - существует суффикс, в котором выполнено f
		* для next - очевидно (умеем отличать next от eventually? на самом деле в асинхранном мире не оперируем фиксацией следующего события - не делаем об этом предположения) - в TLA+ нет
		* always eventually f - на каждом суффиксе f будет выполнена (**fairness**)
		* eventually always  - начиная с некторого момента f выполняется (**eventual consistency** - "сходимость")
	* как описать систему? (Свойства научились выше)
		* свойства вида: Init and always Next (описание графа -  свойство, которое выделяет все допустимые цепочки, в которых в начальной конфигурации выполняется свойство Init и для каждой пары соседних конфигураций выполняется свойство Next (описание перехода между конфигурация; Action в TLA+)), то есть это темпоральная формула
* **но вообще хочется проверять не дизайн алгоритма (какую-то модель, которую мы создаем тупо для проверки дееспособности идеи), а именно реализацию данного дизайна**
	* в первую очередь спецификация не учитывает конкурентность, в т.ч. примитивы синхронизации
	* PlusCal - язык, где уже есть некоторые абстракции для создания многопоточности + трансляция в TLA+
