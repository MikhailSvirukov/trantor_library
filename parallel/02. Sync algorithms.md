Напишем функцию, которая цвеличивает значение на 1
	```
	int my_inc(int *a) {
		*a = *a + 1;
	}
	```
	Помимо сложения есть операции чтения и записи - поэтому есть гонки данных. 
	Если значения локальные - нет проблем, они возникают для разделяемых значений.
	То есть по сути блок read, inc, write
	* есть вариант когда блоки не пересекаются - все ок, ожидаемое 2
	* если после первого чтения сразу идет чтение в другом потоке - проблема - это и есть гонка данных
**Race condition** - состояние, при котором результат вычисления зависит от порядка выполнения операций
* Нарушение согласованности - чтение уже неактуальных данных (какой-то недетерминизм - зависимость не от логики, а от порядка выполнения потоков)
* Такие ошибки сложно воспроизводить и отлавливать данные
Решение проблемы:
* Тип с функциями lock/unlock (блокировка, причем разная гранулярность) - по сути эксклюзивный доступ к критической секции (блокирующая синхронизация)
* Атомарные операции (доступ)
* Неблокирующая синхронизация:
	* *obstraction free* (если есть несколько потоков и для любого из них, и остановке всех прочих, то выбранный будет совершать прогресс - завершение за конечное число шагов) - с блокирующей это не раюотает так как поток, захвативший критическую секцию может не отдать к ней доступ
	* *lock free* - хотя бы один поток совершает прогресс, даже если не все остальные остановлены (`cmp_excg(addr, expected, new)`)
		```
		int my_inc(int *a) {
			old = atomic_read(a)
			cas(a, old, old+1)
		}
		```
		Проблема - между атомарными операциями другой поток может поменять значение между чтением и cmp_exch
	Правильно:
		```
		int my_inc(int *a) {
			while(true) {
				old = atomic_read(a)
				if cas(a, old, old+1) == old {
					break;
				}
			}
		}
		```
	* *wait-free* - все потоки выполняются в конце концов
* Небокирующая синхронизация не всегда быстрее блокирующей - только в особых сценариях

## Блокирующая синхронизация
* **Критическая секция** - последовательность кода, которую может исполнять только один поток
* Огораживается 2 методами: lock() / unlock() (acquire/release)
* Алгоритм с этими методами - алгоритм блокировки
* Ожидаемые свойства:
	* *mutual exclusion* (в секцию заходит один поток)
	* *deadlock freedom* (хотя бы один поток зайдет в секцию; но это только про сам метод lock)
	* *starvation free* (поток, если пытается захватить lock, то он рано или поздно это сделает)
* Проблема в том, что выполнение вышеуказанных требований не влечет отсутствие deadlock при использовании нашей реализации
* Способы:
	* *conditional locking* (сброс lock при попытке захвата и неудаче) + *acquire all lock first* (еще можно sleep с random timeout)
	* создать глобальный счетчик - тогда каждая сущность будет получать уникальное значение из множества с некоторым порядком - при этом сам счетчик тоже разделяемое значение с наибольшим значением (никто другой не получает его при инкременте - так реализовать инкремент)
* *Peterson Lock*:
	* массив флагов + разделяемое значение (свое значение)
	* пишем в разделяемое значение себя + в свое место в массиве true
	* пока другой true и разделяемое значение - сам поток - ожидание
#### Теперь обобщение для n потоков
* *FilterLock* - два массива: уровней (thread с id i находится на уровне k) и victims (какой уровень поток i пытается заблокировать); идея в том, что мы сваливаемся вниз по уровням, емкость которых уменьшается; lock получает тот, который в самом нижнем уровне (емкость уровня - 1)
* *LamportBakery* - два массива - флаги и номерки; пишем в свою ячейку флагов true, а в номерок - наибольший номерок плюс 1; если у кого-то у всех флаги true или false, но номерки больше, то даем доступ (есть проблема со starvation при циклической арифметике - но обычно 64 битов хватает)
