Монитор:
*  cекция входа 
* иногда нужно нужно отпускать блокировку, попав в критическую секцию (mpsc - переплнение очереди или пустая очередь)	
* секция ожидания 
*   взаимодействие: Monitor.enter() => проверка критерия входа в критическую секцию => Monitor.wait() (если провалили условие) => ждем Monitor.pulse_all() (pulseOne())(пнуть всех тех, кто попал в зону ожидания; причем они переместятся в зону входа)
* lost-wake-up
* часто try/finally (потому что после монитор нужно отпускать) или пробуждение по таймауту
RwLock:
* один поток на запись получает доступ к критической секции, то никто больше не получает или много потоков на запись получают доступ
* реализация может быть через монитор; критерий проверки - пара чисел, какие потоки по ролям в критической секции
* **при этом стоит понимать, что блокировка происходит за счет проверки, а не монитора** (по сути методами lock/unlock поверх монитора)
* для честности доступа читателя/писателя - число вызовов acquire и число вызовов release readLock; при unlock читатель смотрит, что число вызовов read lock/unlock сравнялось - тогда pulseAll
* для writeLock - при lock ждем пока не отпустятся все reader (не даем новые readLock)
* но по сути честность - приотеризация записи
ReentrantLock
* тип блокировки, допускающий повторный захват блокировки на одном и том же объекте, если такая блокировка была получена (допускает повторный захват критической секции)
* реализация через монитор - смотрим какой поток держит критическую секцию, если да - увеличиваем количество захватов и входим, если нет - ждем пока все запросы закончат 
Semaphore
* позволяет допускать в критическую секцию фиксированное количество потоков (т.е. секция не совсем критическая)
* через мониторы - просто увеличиваем число, пока не достигли предела
Barier
* работает для заданного числа потоков; метод await - при вызове будет ожидание, пока все потоки не дойдут до точки
* await через барьеры
* простая реализация - просто атомарный декримент - проблема в пробуждене всех
* sense-reverse barier - добавим к простой реализации флаг "закончил ли барьер работу" через фазы 1 и 0 (причем при ожидании флаг берется в поток инвертированным глобальному и threadlocal); оеальное ожидание - по значению флага, а не числа 
* хотим избавиться от атомарной переменной; сделаем дерево, в котором количество листьев - не совсем число потоков, которое будет ожидаться; await делит свой id потока на число детей; у каждого узла свой флаг + число - число детей (по сути сколько раз ждем вызов await на тукещем узле); если дождались детей - инвертируем фазу (флаг) и идем к родителю; по сути ожидание распространяется от листьев к узлу; каждый поток ждет на своей node; ждем на каждой node по сути логарифм (limit) await 
* другой древовидный подход предполагает, что каждый поток будет определять, на каком узле будет ожидать узел (в отличие от предыдущего - тот зависел от порядка); фазу через флаг определяет глобальная переменная, которую меняет только корень - обозначить новую фазу
