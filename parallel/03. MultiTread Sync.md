## Барьеры
реализация параллельного стека
```
push(in)
	i = count
	elem[i] = in
	---------------------- Release barier
	count++
pop
	tmp = count
	count--
	if tmp > 0
	---------------------  Accuire barier
		return elem[tmp]
```
Проблемы - доступ к невалидной памяти (если компилятор (если компилятор не заметит неявную зависимость (строки 3 и 4 в pop)) или процессор (out of order execution, ) переупорядочивает) или потеря (при некотором сценарии) вершины стека при параллельном push/pop

Решается барьером (barier):
* операции, которые (не) могут перескакивать вверх, и которые (не) могут перескакивать вниз (соответственно обозначаются (Store) Load)
* LoadLoad + LoadStore - **Accuire** и LoadStore + StoreStore - **Release**
	```
	read(a)
	write(a)
	----------- barier, b зависит от a (в прошлом примере, i зависит от count)
	read(b)
	write(b) 
	```
	Accuire - то есть сверху барьера может быть перемещено вниз, но не наоборот (по сути, барьер на чтение - можно в прошлом примере в push поменять 2 и 3 строчки местами, но поможет в pop)

Зачем?
У CPU есть какие-то собственные кэши и раздеяемая память; плюс у каждого процессора есть свой WriteBuffer - временное хранилище для изменяемых процессором данными. И процессор спокойно переупорядочивает работу с памятью - некая асинхронная модель взаимодействия с памятью.
Зачем барьер? 
Он убеждается, что вся работа с памятью завершена до барьера + заставляет все процессоры работать с общей памятью. То есть по вызову барьера (инструкция) - процессор делает своеобразный *flush* 
С некоторой точки зрения, можно через lock - медленнее, чем барьеры; атомарные операции - еще хуже, потому что это по сути блокировка некоторого участка shared memory - пока не исчезнет конкурентный доступ

*Как синхронизироваться на произвольном колличестве потоков (в лекции 2 - только для фиксированного количества)?*

* *Для фундаментального решения deadlock - брать или 0 или 1 lock*
* *В целом, сон можно рассматривать как освобождение под полезную работу - не обязательно для именно для нашей программы, может быть и в контексте работы других потоков ОС*

примеры:
* TAS (take and set) - голодание, активное ожидание + аппаратная реализация (постоянная блокировка участка разделяемой памяти) - по сути реализация **spin-lock**
* избавиться от активного ожидания - чтение с барьерами (для проверки значения по адресу) - по сути реализация **mutex**
* **backofflock** - сон на случайное значение из диапазона
* **ArrayLock** - в прошлой реализации существует проблема пересоздания класса задержки, поэтому threadlocal; поток получает номер через атомарный инкремент; потом ждем когда в массиве флаг в собственном номере станет true (причем каждый пытается получить доступ только в своей области памяти); при освобождение ставим следующий флаг в массиве в true, а себя в false; не получается mutual exclusion при числе потоков больше capacity, работает starvation free и deadlock free (если не сделать volatile - есть специфический случай, когда (для 2 потоков) переупорядочатся проставление флагов в массив? разбудит второй и уснет (не проставив себе флаг) и второй поток пройдет цикл и станет первым - а первый проснется и поставит себе false ); короче, решение все равно не решает активное ожидание, но из-за массива нет попыток добиться в одно место - только в свою ячейку (хотя учитывая кэши - это сделает только хуже (инвалидация и обновление по записи в кэш линию и алгоритмам поддержания когерентности))
* на неявных списках - решает mutual exchange, нет голодания, вроде бы норма с памятью? норм, потому что делаем exchange указателей; но крутимся активно на соседнем среде (в threadlocal)
